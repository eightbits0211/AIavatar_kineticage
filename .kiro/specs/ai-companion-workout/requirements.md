# Requirements Document

## Introduction

KineticAge is an AI-powered fitness companion that delivers personalized workout plans, voice and text coaching, motivation, accountability, and progress tracking. The core differentiator is the feeling of having a knowledgeable, supportive training partner available at any time. The workout content (exercise selection, sets, reps, progression) is generated by a deterministic, trainer-approved Rules Engine. The AI layer sits on top to explain choices, motivate the user, answer questions, adjust tone, and provide conversational support. This separation keeps the experience safe, predictable, and explainable while still feeling intelligent and human. Target MVP demo: 17 July 2026.

## Glossary

- **Companion**: The AI coaching layer powered by Claude API (claude-sonnet-4) that explains, motivates, and converses — it never invents exercises or prescribes weights
- **Rules_Engine**: The deterministic, versioned service that generates workouts from user profile + exercise library. Not an LLM. Four stages: Filter → Category → Persona Modifier → Bundle Assembly
- **Exercise_Bundle**: A complete, ready-to-do workout option containing a title, exercises with sets/reps/rest, estimated duration, and calorie burn range. 3-4 generated per cycle, exactly one flagged as recommended
- **Session**: A single workout instance created when a user selects a bundle and begins exercising
- **User_Profile**: All stored user data including physical attributes, goals, equipment, injuries, preferences, persona tags, and calculated metrics
- **Persona_Tags**: An array of automatically-assigned behavioral/attribute labels (2-4 per user) derived from onboarding data and ongoing behavior. Not mutually exclusive.
- **Exercise_Library**: The trainer-approved master library of 80-120 exercises with metadata (muscles, equipment, contraindications, substitution groups, category-specific set/rep defaults)
- **Voice_Pipeline**: The chain of Deepgram (STT) → Claude API → ElevenLabs (TTS streaming) for spoken interaction
- **Text_Chat**: The persistent text interface providing the same AI companion intelligence
- **Talkativeness**: User-controlled setting (Minimal/Balanced/High) governing how often the AI proactively initiates communication
- **In_Session_Verbosity**: Separate setting (Quiet/Standard/Detailed) controlling how much the AI narrates during active exercises
- **Progression_Logic**: Deterministic rules tracking per-exercise completion history to trigger rep/set increases or deloads
- **Backend**: The Node.js + Express server handling auth, rules engine, AI relay, session management, and gamification
- **State_Machine**: The Zustand-managed session state: idle → session_starting → exercise_intro → set_active → set_complete → check_in → rest → session_summary → idle
- **XP**: Experience points awarded for sessions, milestones, and engagement
- **Streak**: Consecutive workout days with a grace-day model (missing 1 scheduled day doesn't break the streak if the user works out the following day)
- **Dashboard**: The home screen showing bundle options, streak, XP/level, history, badges, weekly progress, and calories burned

## Requirements

### Requirement 1: User Onboarding & Profile Setup

**User Story:** As a new user, I want to complete a guided setup so that my workouts are tailored to my body, goals, and constraints.

#### Acceptance Criteria

1. WHEN a new user opens the App for the first time after account creation, THE App SHALL present onboarding as a guided conversation (text by default, with optional voice) rather than a static form, while still collecting all required structured fields
2. THE onboarding flow SHALL collect the following required fields in sequential steps: age (integer, 16-100), height (cm or ft/in, user-selectable unit), weight (kg or lb, user-selectable unit), gender (Male/Female/Other/Prefer not to say), fitness goal (single-select: Strength, Hypertrophy, Mobility, General Fitness, Weight Loss, Home Workout), activity level (Sedentary/Lightly Active/Moderately Active/Very Active), workout location (Gym/Home/Outdoors/Hybrid), available equipment (multi-select from: None, Dumbbells, Barbell, Resistance Bands, Kettlebell, Pull-up Bar, Bench, Machines, Cardio Equipment), injury information (multi-select + free text: None, Knee, Lower Back, Shoulder, Wrist, Ankle, Other), preferred workout duration (15/30/45/60 minutes), and prior program experience ("Have you followed a structured workout program before?" — Yes/No)
3. THE App SHALL display a progress indicator throughout (e.g., "Step 4 of 10") with an estimated completion time of 3-5 minutes
4. THE App SHALL validate all inputs client-side before submission (age 16-100, height/weight within plausible ranges) and prompt confirmation for unusually high/low values
5. IF the user provides an age below 16, THEN THE App SHALL display an age-appropriate message and SHALL NOT permit them to proceed
6. THE injury information field SHALL default to "None" if skipped; all other fields are required
7. IF the user exits onboarding before completion, THEN THE App SHALL save progress and resume from the last completed step on next login
8. WHEN onboarding is submitted, THE Backend SHALL calculate and store BMI, TDEE (daily calorie estimate), MHR (max heart rate), and Target Training Zone, then display a summary with visible "estimate, not medical advice" disclaimers
9. WHEN the user completes onboarding, THE Backend SHALL award 15 XP (one-time onboarding bonus)

### Requirement 2: Calculated Metrics

**User Story:** As a user, I want to see my BMI, calorie needs, max heart rate, and training zone so I have context for my fitness journey.

#### Acceptance Criteria

1. THE Backend SHALL calculate BMI using: weight(kg) / height(m)² and display with WHO category labels (Underweight, Normal, Overweight, Obese) — never as a judgmental label
2. THE Backend SHALL calculate TDEE using the Mifflin-St Jeor equation: Male BMR = (10 × weight_kg) + (6.25 × height_cm) − (5 × age) + 5; Female BMR = (10 × weight_kg) + (6.25 × height_cm) − (5 × age) − 161; Other/Prefer not to say = average of both formulas with a note that it is an estimate
3. THE Backend SHALL apply the activity multiplier to BMR: Sedentary = 1.2, Lightly Active = 1.375, Moderately Active = 1.55, Very Active = 1.725
4. THE App SHALL display TDEE as a range (TDEE ± 5%) with a disclaimer
5. THE Backend SHALL calculate MHR = 220 − age and Target Training Zone = 60% to 80% of MHR, displayed as a bpm range
6. ALL calculated metrics SHALL be recalculated whenever the user updates their profile

### Requirement 3: Persona Assignment

**User Story:** As a user, I want the system to automatically understand my situation and adapt everything accordingly, without me having to manually configure persona settings.

#### Acceptance Criteria

1. WHEN onboarding is completed, THE Backend SHALL run persona assignment rules synchronously and store resulting tags as an array on the user profile BEFORE the user reaches the workout recommendation screen
2. THE system SHALL support the following persona types assigned based on these rules: Complete Beginner (Activity Level = Sedentary + no prior program), Regular Gym-Goer (Activity Level = Moderate/Very Active + Gym + prior program), Weight Loss Seeker (Fitness Goal = Weight Loss), Home Workout User (Location = Home + None/minimal equipment), Office Professional (Sedentary/Light activity + 15 or 30 min duration preference), Injury Recovery User (Injury Information ≠ None), AI Companion Seeker (Talkativeness = High or high engagement behavior in first week), Inconsistent Enthusiast (behavioral — workout completion rate < 50% over rolling 2 weeks, applied after first 2 weeks)
3. A user SHALL carry 2-4 active persona tags simultaneously (tags are NOT mutually exclusive)
4. Persona tags SHALL be re-evaluated weekly (for behavioral personas) or whenever the profile is edited (for attribute-based personas)
5. THE Companion SHALL adapt its tone, proactivity, and message framing based on all active persona tags without altering the underlying workout content
6. THE Rules_Engine SHALL apply persona tags as modifiers to exercise selection, set/rep defaults, and bundle composition using a fixed priority order: Injury exclusions > Equipment/Location filters > Goal-based category logic > Persona-based additions

### Requirement 4: Rules-Based Workout Recommendation Engine

**User Story:** As a user, I want safe, structured, personalized workout options generated by a trainer-approved system, not made up by an AI.

#### Acceptance Criteria

1. THE Workout Recommendation Engine SHALL be a deterministic, versioned rules service — NOT an LLM-based system
2. THE Rules_Engine SHALL run four stages in order: Filter Stage (filter exercise library by location, equipment, injuries), Category Stage (apply goal-mapped workout category rules for set/rep ranges, rest periods, exercise-type ratios), Persona Modifier Stage (apply additive modifiers from active persona tags), Bundle Assembly Stage (assemble 3-4 complete bundles with variety)
3. EACH generation cycle SHALL produce 3-4 distinct Exercise Bundles, differentiated by primary movement focus, duration variant, or intensity
4. EXACTLY one bundle SHALL be flagged is_recommended = true, chosen by scoring against: day's planned focus, muscle groups not recently trained, and user's stated energy/availability
5. THE Rules_Engine SHALL NEVER output a specific weight value — only exercise name, target sets, target rep range, and rest interval
6. FOR every strength-based exercise, THE Companion SHALL include guidance encouraging the user to select a weight that allows completion with good form
7. IF the Filter Stage yields too few eligible exercises, THEN THE Rules_Engine SHALL fall back to a curated "minimal equipment, low-impact" baseline set rather than failing
8. WHEN the AI Service is unavailable, bundles SHALL still display with a generic fallback rationale — generation NEVER blocks on AI availability
9. THE Rules_Engine SHALL apply goal-specific logic: Strength (3-6 reps, 90-180s rest, compound lifts), Hypertrophy (8-12 reps, 60-90s rest, volume focus), Mobility (time-based holds/10-15 reps, 15-30s rest), General Fitness (mixed 8-12 reps, 45-60s rest), Weight Loss (12-20 reps, 15-30s rest, circuit-style), Home Workout (bodyweight-focused, tempo variations, 30-60s rest)

### Requirement 5: Exercise Bundle System

**User Story:** As a user, I want multiple workout options with one clearly recommended, so that I feel in control while still getting guidance.

#### Acceptance Criteria

1. EACH Exercise Bundle SHALL include: title, estimated duration, estimated calorie burn range, exercise list (with sets/reps/rest per exercise), and a one-line AI-generated rationale explaining the recommendation
2. THE App SHALL display the bundle set with the Recommended bundle visually highlighted at the top
3. THE user SHALL be able to view full exercise details for any bundle before selecting
4. WHEN the user selects a bundle (recommended or alternative), THE App SHALL begin a Workout Session
5. IF the user dismisses all bundles, THEN THE Companion SHALL offer to regenerate options (asking about available time or energy level) and the Rules_Engine SHALL re-run with adjusted constraints
6. FOR returning users, bundles SHALL account for recent history (muscle group rotation, progression flags, missed sessions)
7. THE AI-generated rationale text SHALL be generated using only structured data from the Rules_Engine — the AI never invents exercise content, only explains it

### Requirement 6: Exercise Library

**User Story:** As a user, I want access to a comprehensive, trainer-approved exercise database so I understand what I'm doing and why.

#### Acceptance Criteria

1. THE Exercise_Library SHALL contain 80-120 trainer-approved exercises covering all Workout Categories, all equipment/location combinations, and common substitution groups
2. EACH exercise record SHALL include: exercise_id, name, category_tags (applicable workout categories), muscle_groups (primary + secondary), equipment_required, location_compatible (Gym/Home/Outdoors/Hybrid flags), contraindications (injury tags), substitution_group (linking substitutable exercises), default_set_rep_range (per-category), instructions_text, difficulty_level (Beginner/Intermediate/Advanced), and image_url
3. WHEN a user asks "how do I do this" (text or voice), THE Companion SHALL provide a structured explanation from the exercise library's instructions_text — expanded conversationally but NEVER inventing new exercise instructions
4. Exercise explanations SHALL include: setup, execution cues, common mistakes, and "choose a weight you can control with good form" guidance for resistance exercises
5. THE Exercise_Library SHALL be seeded from a structured JSON reviewed by a trainer/SME

### Requirement 7: Workout Session

**User Story:** As a user, I want to be guided through my workout hands-free, so that I can train without constantly checking my phone.

#### Acceptance Criteria

1. WHEN a user selects a bundle, THE Backend SHALL create a new Session record (status = in_progress) and display the first exercise with sets, reps, rest interval, and a short instructional reference
2. THE user SHALL be able to interact via voice (press-to-talk) at any point — asking "what's next", "how do I do this", "skip this one" without requiring manual navigation
3. AFTER completing a set, THE user SHALL mark progress (tap "done" or say "done" via voice); rest timer begins automatically if rest_seconds > 0
4. THE user SHALL mark each exercise as complete, skipped, or note "felt hard"/"felt easy" upon finishing
5. WHEN all exercises are completed, THE session SHALL be marked status = full; XP awarded per gamification rules; AI provides a post-workout summary
6. IF the user exits mid-session and has completed ≥50% of exercises, THEN the session SHALL be marked status = partial with partial XP awarded
7. IF the user exits mid-session with <50% completion, THEN the session SHALL be marked status = abandoned (no XP, but progress saved); IF the user returns within 30 minutes, THE App SHALL offer to resume the session from where they left off
8. THE State_Machine SHALL follow: idle → session_starting → exercise_intro → set_active → set_complete → check_in → rest → (loop or summary) → idle

### Requirement 8: Progression Logic

**User Story:** As a user, I want my workouts to gradually get harder as I improve, so that I keep making progress.

#### Acceptance Criteria

1. THE Backend SHALL track completion history per exercise (or substitution group) across sessions
2. IF a user completes the top of the prescribed rep range for an exercise across 2 consecutive sessions, THEN THE Rules_Engine SHALL increase the prescribed rep range or add a set in the next bundle containing that exercise
3. WHEN rep progression criteria are met, THE Companion SHALL suggest — in conversational terms — that the user consider a slightly heavier weight next time, WITHOUT specifying an amount
4. IF a user marks an exercise as "felt hard" or skips it 2+ times, THEN THE Rules_Engine SHALL deprioritize it in favor of a substitution-group alternative in subsequent bundles
5. Progression changes SHALL be deterministic and rules-based, NEVER generated by the AI

### Requirement 9: Voice Input

**User Story:** As a user, I want to speak to the companion naturally so that I can interact hands-free while exercising.

#### Acceptance Criteria

1. THE App SHALL provide a press-to-talk voice interface (always-on/wake-word listening is out of scope for MVP)
2. WHEN the user activates voice input, THE Voice_Pipeline SHALL capture audio and send to Deepgram for STT within 500ms of activation
3. WHEN Deepgram returns a transcription, THE Backend SHALL forward to Claude API with current context (profile, persona tags, session state, recent conversation) within 1 second
4. IF Deepgram fails within 5 seconds, THEN THE Backend SHALL retry once with 2-second delay
5. IF voice transcription fails after retry, THEN THE App SHALL prompt the user to try again or switch to text, preserving session state
6. DURING an active session, voice SHALL be the primary interaction mode — users can ask "how many reps left", "what's next", "how do I do this exercise" without touching the screen
7. Voice responses during active sets SHALL be kept concise (1-2 sentences); longer explanations offered only between sets or on request
8. IF voice command is ambiguous, THE Companion SHALL ask a brief clarifying question rather than guessing at an action
9. IF transcription confidence is low (background noise), THE Companion SHALL acknowledge uncertainty ("Sorry, I didn't catch that — could you repeat?")

### Requirement 10: Voice Output

**User Story:** As a user, I want the companion to speak to me aloud so I can follow along without looking at the screen.

#### Acceptance Criteria

1. WHEN Claude returns a response, THE Backend SHALL stream to ElevenLabs for TTS conversion
2. THE Voice_Pipeline SHALL use ElevenLabs streaming so first audio chunk plays within 2 seconds
3. THE Companion SHALL use a single consistent ElevenLabs voice ID across all sessions
4. IF ElevenLabs fails within 5 seconds, THEN retry once; if still fails, display text and notify voice is temporarily unavailable
5. WHILE voice plays, corresponding text SHALL display on screen simultaneously
6. IF the user initiates new voice input while the Companion is speaking, THE Voice_Pipeline SHALL interrupt playback and process new input
7. THE response text SHALL also be displayed in the chat interface for accessibility and reference

### Requirement 11: Text Chat

**User Story:** As a user, I want a persistent text chat so I can ask questions, request changes, or interact with the companion anytime.

#### Acceptance Criteria

1. THE App SHALL provide a persistent chat interface accessible from any screen
2. THE Text_Chat SHALL use the same Claude API system prompt and context as voice, providing identical companion behavior
3. Chat history SHALL be retained per user and used as short-term context for the AI (with appropriate context window management)
4. WHEN a user switches between voice and text mid-session, THE Companion SHALL continue without interruption or context loss
5. THE Text_Chat SHALL display messages with clear visual distinction between user and companion, minimum 16sp font
6. IF Claude fails to respond within 10 seconds, retry once; if still fails, display error message

### Requirement 12: AI Companion Behavior

**User Story:** As a user, I want an AI companion that explains my workouts, motivates me, and adapts its personality to my preferences.

#### Acceptance Criteria

1. THE Companion SHALL NEVER invent exercises, prescribe specific weights, or generate workout content — it only explains, motivates, and answers questions using data from the Rules_Engine and Exercise_Library
2. THE Companion SHALL include a standard disclaimer that KineticAge is not a substitute for medical or physical therapy advice (for Injury Recovery users: once per week, not every session)
3. AFTER each workout, THE Companion SHALL provide a short summary: exercises completed, any progression flags triggered, streak status, and forward-looking encouragement
4. WEEKLY, THE Companion SHALL provide a longer recap referencing the Dashboard's weekly progress data
5. Motivational messaging SHALL be template-driven with AI-personalized phrasing, tied to real events: streaks, personal bests, returning after a break, milestone workout counts
6. THE Companion's tone SHALL adapt to active personas: educational for Complete Beginner, peer-like for Regular Gym-Goer, avoids appearance-based language for Weight Loss Seeker
7. IF a user's voice query maps to a structured action (e.g., "skip this exercise", "start my workout"), THE Companion SHALL return both a conversational response AND a structured action intent for the frontend to execute

### Requirement 13: Adjustable Talkativeness

**User Story:** As a user, I want to control how often the AI talks to me, so the experience matches my preference.

#### Acceptance Criteria

1. DURING onboarding (and editable anytime in Settings), THE App SHALL allow users to set talkativeness level: Minimal (AI only responds when user initiates, no proactive messages), Balanced (workout reminders, post-workout summaries, occasional motivational check-ins, max ~1/day), or High (proactive motivational messages, tips, streak check-ins, up to several times/day)
2. DURING an active workout, a SEPARATE in-session verbosity setting (Quiet/Standard/Detailed) SHALL control how much the AI narrates each exercise, independent of the overall talkativeness setting
3. WHEN talkativeness is set to Minimal, THE Companion SHALL NOT send any proactive messages — only responding to user-initiated interactions
4. WHEN talkativeness is set to High and a day passes without a workout, THE Companion MAY send a proactive encouraging check-in (subject to a sensible daily cap)
5. WHEN in-session verbosity is Quiet, THE Companion SHALL provide exercise name and rep count only — no full instructions unless requested

### Requirement 14: Safety and Pain Handling

**User Story:** As a user, I want the companion to stop if I report pain and never push unsafe exercises.

#### Acceptance Criteria

1. WHEN the user reports pain, THE Companion SHALL stop current exercise guidance and acknowledge the pain
2. THE Companion SHALL suggest rest and offer to skip to the next exercise or end the session
3. THE Companion SHALL NEVER encourage pushing through reported pain
4. Pain events SHALL be recorded including exercise, body area, and timestamp
5. THE Rules_Engine SHALL reference pain history to exclude/substitute exercises in future bundles
6. IF a user asks a medical question, THE Companion SHALL decline, state it cannot provide medical advice, and direct to a professional
7. THE Companion SHALL NOT provide medical diagnoses, treatment recommendations, or physiotherapy advice

### Requirement 15: Gamification — Streaks

**User Story:** As a user, I want streak tracking with a grace-day model that doesn't punish occasional misses.

#### Acceptance Criteria

1. WHEN a user completes at least one session on a scheduled day, THE Backend SHALL mark it as an active streak day
2. THE streak SHALL use a grace-day model: missing a single scheduled day does NOT break the streak if the user completes a workout the following day (1 grace day per week, configurable)
3. THE Backend SHALL store current_streak and longest_streak
4. WHEN a streak reaches milestones (7, 30 days), THE Backend SHALL award bonus XP and THE Companion SHALL celebrate
5. IF a streak breaks, THE Companion SHALL use "streak recovery" framing (restart, not loss/shame) — especially for the Inconsistent Enthusiast persona
6. THE "Comeback" achievement SHALL reward returning after a 7+ day gap (rewards re-engagement, not punishes absence)

### Requirement 16: Gamification — XP System

**User Story:** As a user, I want to earn XP for my activity so I can see my engagement grow.

#### Acceptance Criteria

1. THE Backend SHALL award XP per these rules: full workout bundle completed = 50 XP, partial workout (50%+ exercises) = 25 XP, progression milestone triggered = 20 XP bonus, 7-day streak = 100 XP bonus, 30-day streak = 500 XP bonus, first workout after onboarding = 30 XP (one-time), completing onboarding fully = 15 XP (one-time)
2. Levels SHALL be defined as: Level N requires N × 200 XP (escalating curve)
3. WHEN a user levels up, THE Companion SHALL celebrate with a short message and THE App SHALL show a visual animation
4. THE App SHALL display total XP, current level, and progress toward next level on the Dashboard

### Requirement 17: Gamification — Achievements

**User Story:** As a user, I want achievement badges that celebrate milestones.

#### Acceptance Criteria

1. THE App SHALL support the following MVP badges: "First Step" (complete first workout), "Consistency Starter" (3 workouts in 7 days), "Week Warrior" (7-day streak), "Momentum" (30-day streak), "Comeback" (return after 7+ day gap), "Leveling Up" (trigger first progression milestone), "Goal Getter" (10 workouts within selected goal category)
2. Badge criteria SHALL be evaluated after each relevant event (session completion, streak update, progression milestone)
3. Newly earned badges SHALL be surfaced with a celebratory UI moment and marked "new" until acknowledged
4. THE Dashboard SHALL display earned and locked badges in a grid

### Requirement 18: Gamification — Progress Tracking

**User Story:** As a user, I want to see my progress so I stay motivated.

#### Acceptance Criteria

1. THE Dashboard SHALL display: workout history (chronological list with date, bundle name, duration, completion status), current streak + longest streak, XP total and level progress, achievement badges, weekly progress (workouts completed vs planned), and estimated calories burned for the week
2. Dashboard data SHALL be cached client-side and refreshed on pull-to-refresh or after completing a session
3. WEEKLY, an aggregation job SHALL compute the weekly progress summary for Dashboard display
4. THE App SHALL display per-exercise progress showing how rep counts have changed over time

### Requirement 19: Daily Check-Ins

**User Story:** As a user, I want brief check-ins that help the AI understand my recovery and energy.

#### Acceptance Criteria

1. WHEN the user opens the App on a non-workout day (and talkativeness ≥ Balanced), THE Companion SHALL initiate a brief check-in about energy and soreness
2. THE check-in SHALL collect: energy level and any soreness from previous workout
3. WHEN soreness is reported, THE Rules_Engine SHALL factor it into next workout's exercise selection
4. THE Backend SHALL store check-in data and make it available to the Rules_Engine
5. Daily check-ins SHALL award 10 XP (mapped to existing XP system)
6. Check-ins SHALL NOT be initiated if talkativeness is set to Minimal

### Requirement 20: Authentication and Data Persistence

**User Story:** As a user, I want my data saved securely so my progress persists.

#### Acceptance Criteria

1. THE App SHALL support account creation via email/password or social sign-in using Firebase Auth
2. THE Backend SHALL authenticate all API requests via Firebase JWT verification
3. THE Backend SHALL store all data (profile, sessions, progression history, gamification) in MongoDB associated with the authenticated user ID
4. IF an API call to Claude, Deepgram, or ElevenLabs fails, THE Backend SHALL retry up to 3 times with exponential backoff before returning an error
5. ALL third-party API keys SHALL remain server-side and never be exposed to the mobile client
6. IF all retries are exhausted, THE Backend SHALL return a structured error indicating which service is unavailable

### Requirement 21: Companion Personality

**User Story:** As a user, I want the companion to feel consistent, warm, and knowledgeable.

#### Acceptance Criteria

1. THE Companion SHALL have a defined name used consistently across all interactions
2. THE Companion's personality SHALL be defined via the Claude API system prompt, persisting across sessions
3. THE Companion SHALL use warm, patient, encouraging communication with simple vocabulary
4. THE Companion SHALL NOT use fitness jargon without plain-language explanation
5. THE Companion SHALL address the user by their preferred name
6. THE Companion SHALL adapt tone based on active persona tags (Section 3.3 of PRD)
